# Plan Fix 39 — Bootstrap Leftover Cleanup

**Status:** Complete (single PR)
**Owner:** adi-hw
**Effort:** 1 PR
**Related canon clauses:** §1 (greenfield discipline), §28 (Plan Fix 33 leftovers)
**Triggering audit:** End-to-end local bootstrap testing on 2026-05-13 — two issues
surfaced after Plan Fix 38 was already dispatched:
1. `ADMIN_PASSWORD` (setup.ts env generation + seed-admin-user.ts reader) vs
   `DEFAULT_ADMIN_PASSWORD` (seed migration + generate-local-dev-secrets.ts) naming drift
2. Plan Fix 33 admin bypass removal left one unused `ctx` parameter in
   `authorization.service.ts` — TS6133 fires at build time

## Context

### Issue 1: env var name mismatch

Two scripts generated or read `ADMIN_PASSWORD`; two other files used `DEFAULT_ADMIN_PASSWORD`.
The split meant `npm run setup` would write the wrong key into `.env`, causing the instance
seed migration (`1818000000000-seed-admin-user.ts`) to crash with:

```
Error: DEFAULT_ADMIN_PASSWORD environment variable is required to seed the bootstrap admin user.
```

The drift arose because `scripts/setup.ts` and `scripts/seed-admin-user.ts` were written
independently of the migration. `generate-local-dev-secrets.ts` and the migration both used
`DEFAULT_ADMIN_PASSWORD` — the canonical name matches the migration's intent ("the default
bootstrap admin's password").

### Issue 2: unused `ctx` in `maskCollectionRecord`

Plan Fix 33 removed 10 admin-bypass sites from `authorization.service.ts`. In
`maskCollectionRecord`, the bypass body was `if (ctx.isAdmin) return record;` — once
removed, `ctx` was no longer referenced in the method body. The parameter was retained for
interface/caller compatibility but TypeScript's `no-unused-vars` rule (ESLint
`@typescript-eslint/no-unused-vars: error`) fires because it's not underscore-prefixed.

## Fix 1: env var name alignment

**Canonical name: `DEFAULT_ADMIN_PASSWORD`**

Rationale: the seed migration (`1818000000000-seed-admin-user.ts`) is the authoritative
bootstrap point and uses `DEFAULT_ADMIN_PASSWORD` in three places (comment, env read,
length check). `scripts/generate-local-dev-secrets.ts` — the parallel secrets generator —
also uses `DEFAULT_ADMIN_PASSWORD`. The "DEFAULT_" prefix accurately conveys "the password
for the default/bootstrap admin user". These two canonical references outweigh the two
wrong consumers.

Files changed:

- `scripts/setup.ts`: env template line changed from `ADMIN_PASSWORD=${adminPassword}` to
  `DEFAULT_ADMIN_PASSWORD=${adminPassword}`.
- `scripts/seed-admin-user.ts`: `requireEnv('ADMIN_PASSWORD')` changed to
  `requireEnv('DEFAULT_ADMIN_PASSWORD')`.

`migrations/instance/1818000000000-seed-admin-user.ts` is unchanged — it was already
canonical.

Existing `.env` files generated by Plan Fix 35–38 setups carry the old `ADMIN_PASSWORD`
key. Users should run `npm run setup:fresh` or manually rename the key in their `.env` to
`DEFAULT_ADMIN_PASSWORD`.

## Fix 2: unused `ctx` parameters

One method in `libs/authorization/src/lib/authorization.service.ts` had its `ctx`
parameter orphaned by Plan Fix 33's bypass removal:

| Method | Line | Change |
|---|---|---|
| `maskCollectionRecord` | 896 | `ctx` → `_ctx` |

The underscore prefix matches the ESLint `argsIgnorePattern: '^_'` convention and signals
intentional non-use without changing the method signature (callers are unaffected; the
deprecated `maskRecord` wrapper continues to pass `ctx` through).

`libs/auth-guard/src/lib/permissions.guard.ts` was audited separately. The guard's bypass
removal in Plan Fix 33 operated on the `canActivate(context: ExecutionContext)` method
where the bypass was a check on `request.user.isAdmin` inside the method body — no `ctx`
parameter existed to orphan.
