# Plan Fix 38 — Migration Runner Hardening + Setup UX

**Status:** Complete (single PR)
**Owner:** adi-hw
**Effort:** 1 PR
**Related canon clauses:** §1 (greenfield discipline; broken bootstrap is a quality issue)
**Triggering audit:** End-to-end bootstrap testing on 2026-05-13 surfaced 5 distinct issues. Agents who shipped Plan Fix 27, 35, 36 verified TypeScript compile + scanner pass but never ran migrations against a live Postgres. This PR closes the gaps.

## Context

Running `npm run setup` end-to-end failed in five distinct ways:

1. `migrationsTransactionMode: 'all'` (TypeORM default) forbids per-migration transaction override — CONCURRENTLY-needing migrations rejected at runtime
2. Migration imports `@hubblewave/instance-db` via the package barrel — unresolved at TypeORM CLI ts-node time (no path mapping registered)
3. Co-located `.spec.ts` files in `migrations/instance/` — picked up by the `*.ts` migrations glob, broken imports cascade
4. Stale `.env` from prior setup runs — setup silently loaded it and users hit missing-var errors with no explanation
5. Stale Postgres volume — password mismatch with current `.env` produces cryptic 28P01 error with no recovery hint

## What landed

### Issue 1 — `migrationsTransactionMode: 'each'` (both datasource configs)

**Files:** `scripts/datasource-instance.ts`, `scripts/datasource-control-plane.ts`

Added `migrationsTransactionMode: 'each'` to both DataSource configs. With `'each'`, every migration runs in its own transaction by default; migrations that declare `public transaction = false;` (required for `CREATE INDEX CONCURRENTLY` — W6.B, F136 PR-3) are honored instead of rejected.

Also updated the `migrations` array in each config to include a negative glob entry (`!...*.spec.ts`) as defense-in-depth against Issue 3.

### Issue 2 — `tsconfig-paths/register` for `@hubblewave/*` imports in migrations

**Files:** `scripts/tsconfig.migration.json`, `package.json`

`tsconfig-paths` was already in devDependencies. Added two layers:

1. `"require": ["tsconfig-paths/register"]` in the `ts-node` config section of `scripts/tsconfig.migration.json`. This registers path mappings when ts-node loads the tsconfig.
2. `-r tsconfig-paths/register` flag on both `migration:run:instance` and `migration:run:control-plane` scripts in `package.json`. Belt-and-suspenders: the flag fires before ts-node reads the config, guaranteeing path resolution even if TypeORM CLI invokes ts-node in a way that doesn't process the `ts-node.require` JSON field.

The `paths` mappings themselves are inherited from `"extends": "../tsconfig.base.json"` which already defines all `@hubblewave/*` mappings relative to the repo root.

### Issue 3 — Spec file removed from migrations directory

**Files:** `migrations/instance/1930900000001-backfill-audit-log-hash-chain.spec.ts` (deleted via `git mv`), `libs/instance-db/src/lib/migrations/1930900000001-backfill-audit-log-hash-chain.spec.ts` (new location)

Used `git mv` to preserve history. Updated the spec's imports from `@hubblewave/instance-db` (circular when spec lives inside the library) to relative imports:
- `AuditLog` from `'../entities'`
- `buildAuditLogHash`, `buildAuditLogHashPayload` from `'../audit-log-hash'`

The spec is now covered by `libs/instance-db`'s existing `tsconfig.spec.json` (`src/**/*.spec.ts` glob) and jest config.

The negative glob entry in both datasource configs (`!...*.spec.ts`) is retained as defense-in-depth — if another spec is co-located in the future, it won't break the migration run.

### Issue 4 — Stale `.env` detection

**File:** `scripts/setup.ts`

Added `REQUIRED_ENV_VARS` constant listing all vars generated by the canonical setup. When an existing `.env` is found, `setupEnvFiles` now:
1. Loads it as before
2. Checks for missing vars
3. If any are absent: emits a `[WARN]` listing the missing vars + instructs the user to run `npm run setup:fresh` or add them manually
4. Calls `process.exit(1)` so the failure is explicit rather than propagating to mysterious downstream errors

### Issue 5 — Postgres 28P01 credential mismatch detection

**File:** `scripts/setup.ts`

Added `verifyPostgresCredentials()` called immediately after `pg_isready` confirms the server is up. The function attempts a real `pg.Client` connection:
- `28P01` (password authentication failed): emits a clear 4-line error naming the root cause (stale volume), the fix (`docker-compose down -v`), and the convenience alias (`npm run setup:fresh`), then exits.
- `3D000` (database does not exist): silently returns — expected on first run before migrations create the database.
- Other errors: logged and process exits.

### Bonus — `setup:fresh` script

**File:** `package.json`

```
"setup:fresh": "docker-compose down -v && node -e \"...\" && npm run setup"
```

Runs `docker-compose down -v` (drops data volumes), deletes `.env` via a cross-platform Node one-liner (no `rimraf` dependency needed), then runs `npm run setup` for a clean-slate bootstrap. Safe to run whenever the user wants to reset the dev environment entirely.

## Acceptance

- All 97+ migrations execute including the CONCURRENTLY-needing W6.B + F136 PR-3 migrations
- `npm run migration:run:instance` resolves `@hubblewave/instance-db` imports at runtime
- Spec files in `migrations/instance/` (if any are added in future) are skipped by the migrations glob
- Setup exits with a clear message when `.env` is missing required vars
- Setup exits with a clear message and recovery hint when Postgres rejects credentials (28P01)
- `npm run setup:fresh` drops volumes + env + re-runs setup cleanly
- All 9 architectural scanners green
